# Phase 07.4: Guided Remediation Workflow - Context

**Gathered:** 2026-05-16
**Status:** Ready for planning

<domain>
## Phase Boundary

Turn completed non-pass validation results into a read-only, run-scoped
remediation queue inside the private validation workbench. The queue should
help maintainers see what failed, where to go next, and which existing
private surface to inspect, while keeping validation evidence immutable and
avoiding any browser-side writeback or new public route.

</domain>

<decisions>
## Implementation Decisions

### Queue scope and data source
- **D-01:** The queue is derived from the currently selected completed run's
  `ValidationRunDetail.results`; there is no new persisted remediation store,
  no backend endpoint, and no cross-run backlog.
- **D-02:** Only `weakSupport`, `refused`, `failed`, and `error` rows are
  emitted. `pass` rows stay in the results table but do not become
  remediation items.
- **D-03:** Item identity is the run-local `resultId` and item order follows
  the immutable result array order so the queue stays aligned with the result
  table.

### Triage vocabulary
- **D-04:** Each item gets a deterministic likely-cause label and review
  status based only on the result outcome. Use a small local mapper rather
  than extending source inventory validation labels.
- **D-05:** The status vocabulary should stay readable and non-mutating, for
  example `Needs source follow-up`, `Needs validation review`,
  `Needs investigation`, and `Needs diagnostics`; it should not imply a
  close/resolve state because 07.4 is review-only.
- **D-06:** The queue should prefer `retrievedSourceIds[0]` as source context
  when present and keep additional retrieved ids or citations as supporting
  evidence, not as alternate identities.

### Destination surface and links
- **D-07:** When a result has source context and the outcome is
  `weakSupport`, the primary destination is the existing private source
  detail page at `/maintainer/sources/:id?preset=validation-follow-up`.
- **D-08:** For `refused`, `failed`, and `error` results, the primary
  destination is the existing result-detail surface in the validation
  workbench (the Inspect overlay), not a new remediation route.
- **D-09:** No new route section, public page, or browser-side remediation
  writeback is introduced; the workbench keeps using the current private
  maintainer navigation and inspect patterns.

### Workbench placement
- **D-10:** The remediation queue lives inline in the validation workbench,
  below run summary/results and above run history, so the evidence surface
  stays first and the triage surface stays adjacent.
- **D-11:** Completed runs with zero non-pass items show an empty-state
  message, while queued/processing or partial/stale runs reuse the existing
  validation alert or partial-state handling rather than inventing a second
  loading model.

### Testing and module shape
- **D-12:** Keep the triage mapping pure and validation-local so it can be
  covered with co-located Vitest/RTL tests in the validation feature folder.
- **D-13:** Do not change source inventory validation helper vocabularies or
  source-history readback behavior for this phase; those surfaces remain
  read-only consumers of existing validation evidence.

### the agent's Discretion
- Exact file names for the queue component or mapper are flexible as long as
  the validation workbench stays the only entry point and the queue remains
  derived from `ValidationRunDetail`.
- Whether the queue renders as a compact table or stacked cards can be
  finalized in planning, provided the order and fields stay stable.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `src/components/modules/MaintainerDashboard/validation/ValidationWorkbench.tsx`:
  already loads validation sets, run history, the active run, and the result
  overlay, so it is the natural host for a derived remediation queue.
- `src/components/modules/MaintainerDashboard/shared/maintainerDashboardShared.tsx`:
  already provides `validationAlertForRun`, `validationDetailAlert`, and the
  result overlay that expose stale, partial, and detailed evidence states.
- `src/components/modules/MaintainerDashboard/sources/SourceDetailPanel.tsx`
  and `SourceValidationEvidence`: already expose the
  `validation-follow-up` source-detail hop.
- `src/components/modules/MaintainerDashboard/shared/routing.ts` and
  `AdminPortalShell.tsx`: already define the private `/maintainer/validation`
  and `/maintainer/sources/:id` navigation boundaries.
- `src/lib/maintainer/api.ts` and `src/lib/maintainer/validation-api.ts`:
  already expose `ValidationRunDetail`, `ValidationResult`, and the fetch
  helpers needed to derive the queue from the existing run-detail payload.

### Established Patterns
- Validation runs are immutable protected records, while source-facing
  history is a separate readback trail.
- Browser code reads validation evidence; it does not manufacture or persist
  new history rows.
- Validation workbench is the launch and run-detail surface; source detail is
  the follow-up surface for source-context investigations.
- Typed union states and local alert helpers are the existing pattern for
  partial, stale, empty, and outage handling.

### Integration Points
- `src/components/modules/MaintainerDashboard/MaintainerDashboard.tsx` route
  wiring already branches to the validation workbench and source detail.
- `src/components/modules/MaintainerDashboard/validation/ValidationWorkbench.tsx`
  is the correct render point for the new remediation section.
- `src/components/modules/MaintainerDashboard/MaintainerDashboard.test.tsx`
  already exercises the validation workbench path and can be extended with a
  richer mixed-outcome seed for queue coverage.
- `ValidationResultOverlay` already provides the run-level detail surface that
  can serve as the primary destination for refused, failed, and error items.

</code_context>

<specifics>
## Specific Ideas

- Keep the queue visually adjacent to the evidence it summarizes, likely
  between the result table and immutable run history.
- The current validation workbench fixture already covers the page and overlay
  path, but it will need a richer mixed-outcome seed before the remediation
  queue can be verified well.

</specifics>

<deferred>
## Deferred Ideas

- Helper actions such as open source, inspect citations, rerun validation,
  queue re-ingest, mark expected refusal, or AI suggestions belong to later
  phases and should stay out of 07.4.
- Any mutable close / resolve state for remediation items is future work, not
  part of the review-only queue.

</deferred>
