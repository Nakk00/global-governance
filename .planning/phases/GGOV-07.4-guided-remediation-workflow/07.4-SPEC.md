# Phase 07.4: Guided Remediation Workflow - Specification

**Created:** 2026-05-16
**Ambiguity score:** 0.14 (gate: <= 0.20)
**Requirements:** 4 locked

## Goal

Completed non-pass validation results become a guided remediation queue inside the private validation workbench so maintainers can see what failed, where to go next, and what kind of repair path is appropriate without changing immutable validation evidence.

## Background

Today, `src/components/modules/MaintainerDashboard/validation/ValidationWorkbench.tsx` already loads validation sets, immutable run history, and full run detail for completed validation runs. The run detail already exposes the result metadata needed for triage, including outcome, expected state, actual state, answer preview, retrieved source ids, citation ids, support score, latency, and notes. `backend/validation/repository.py` already creates immutable validation runs and appends terminal source-facing history after completion, and `backend/sources/repositories/mappers.py` already reads `latestValidationOutcome` and `validationHistory` back into source detail and inventory views.

What does not exist yet is a remediation queue or item model. Maintainers can inspect the result table and source detail today, but they still have to infer the likely failure type, the safest next surface, and the right review status by reading raw validation output themselves. This phase adds that guided triage layer inside the private maintainer workflow without introducing repair actions, AI suggestions, or any browser-side writeback path.

## Requirements

1. **Remediation itemization**: Each completed non-pass validation result becomes a remediation item with stable identity, run/question context, source context when available, and a visible workflow status.
   - Current: The validation workbench renders run detail, result rows, and a result overlay, but there is no remediation item model or queue.
   - Target: A terminal validation run with `weakSupport`, `refused`, `failed`, or `error` results renders one remediation item per non-pass result, while `pass` results do not produce items.
   - Acceptance: A seeded run with mixed outcomes shows zero remediation items before terminal completion, then one item per non-pass result after completion, with no item created for `pass` results.

2. **Deterministic triage metadata**: Each remediation item shows a deterministic likely-cause label, destination surface, next action, and review status derived from the completed validation result.
   - Current: Maintainers can infer the issue from raw outcome metadata, but the workbench does not summarize the likely failure type or the next hop.
   - Target: The remediation queue surfaces triage metadata that is stable for the same completed run and readable without opening the result overlay.
   - Acceptance: Reloading the same completed run shows the same item count, same likely-cause labels, same destination surface labels, same next-action labels, and the same workflow status values.

3. **Evidence separation**: Remediation items stay review-only and do not mutate validation evidence or source-facing history.
   - Current: Validation runs are immutable, the source-history bridge already writes terminal evidence, and there is no remediation workflow yet.
   - Target: Listing or triaging remediation items does not change `validation_runs`, `validation_results`, or `source_validation_runs`, and the queue does not introduce browser-side writeback to those records.
   - Acceptance: Validation detail and source-history readbacks remain unchanged after opening or refreshing the remediation queue, and the remediation surface only reads existing evidence.

4. **Private routing**: Each remediation item deep-links into an existing private maintainer surface relevant to its classification instead of creating a separate public or learner-facing repair page.
   - Current: Source detail, validation detail, and evidence inspection surfaces already exist inside the private maintainer shell.
   - Target: The remediation queue gives maintainers a direct next hop to the right existing private surface for each item.
   - Acceptance: Every remediation item exposes at least one working deep link to an existing private maintainer surface, and the feature is not reachable from public routes.

## Boundaries

**In scope:**
- A remediation section or queue inside the private validation workbench
- Deterministic classification and routing metadata for completed non-pass validation results
- Visible workflow status and stable item identity for the remediation queue
- Deep links into existing private maintainer surfaces for review and follow-up

**Out of scope:**
- Helper actions such as open source, inspect citations, queue re-ingest, mark expected refusal, or rerun validation - those belong to Phase 07.5
- AI-generated diagnosis, draft notes, or suggestion copy - those belong to Phase 07.6
- Any automatic source mutation, validation rewrite, or browser-side writeback path
- Run-launch failures or pre-completion infrastructure outages - this phase only materializes items from completed validation result data
- Public or learner-facing routes, dashboards, or repair pages

## Constraints

- Validation runs remain immutable, and remediation must stay separate from the evidence record.
- Only completed validation runs produce remediation items; in-flight or aborted runs do not.
- The remediation workflow stays inside the private maintainer boundary and reuses existing source, validation, and evidence surfaces.
- No helper-action, rerun-closeout, or AI suggestion contract is introduced in this phase.

## Acceptance Criteria

- [ ] A completed validation run with non-pass results renders one remediation item per non-pass result in the private validation workbench.
- [ ] `pass` results do not create remediation items.
- [ ] Each remediation item shows outcome, likely cause, destination surface, next action, and workflow status.
- [ ] Reloading the same completed run preserves the same item count, stable labels, and stable status values.
- [ ] Each remediation item offers at least one working deep link to an existing private maintainer surface relevant to its classification.
- [ ] Opening or refreshing the remediation queue does not change validation run detail or source-history data.
