## Epic Review

### Outcome

- Epic 2 is complete: all 3 stories are done in the sprint ledger.
- The UN module now covers the shell, organ selection, and responsive comparison polish.
- The work stays inside the single-page anchor-navigation model and keeps the `un-command-center` identity intact.
- Story records report successful validation with `pnpm format`, `pnpm lint`, `pnpm typecheck`, `pnpm build`, and Playwright smoke coverage.

### Follow-Through from Epic 1

- The shell and navigation contract from Epic 1 was reused instead of replaced.
- Summary-first reading, focus restoration, reduced motion, and no-horizontal-scroll expectations carried forward cleanly.
- Shared editorial tokens and authored content modules stayed the source of truth.
- Playwright coverage expanded from baseline shell smoke into module-specific regression checks.

### What Went Well

- The feature grew in thin layers: shell first, then organ exploration, then responsive comparison polish.
- The organ content stayed structured in `src/data/sections/un-command-center.ts`, which kept the chapter voice reviewable and consistent.
- The module stayed intentionally bounded to a small set of organs, which made the comparison story clear instead of encyclopedic.
- Responsive layout work was treated as part of the feature, not as a cleanup step after review.
- The smoke suite focused on rendered behavior, keyboard paths, reduced motion, and direct hash entry, which matched the actual risk.

### Friction and Lessons

- Selector semantics must be explicit. Review feedback correctly caught a control-pattern issue before the epic closed.
- Feature state needs safe defaults. An empty or delayed dataset should not be able to produce an undefined selected organ.
- Responsive comparison needs distinct reading modes for desktop and smaller screens, not just a generic grid split.
- Fixed delays in tests are a flake risk. Geometry and rendered-state assertions are more reliable for this module.
- Closeout needs a status-sync step. The story record and sprint-status can drift if the handoff checklist does not update both.
- The review loop was useful because it found semantic, state, and test issues that were worth fixing before the module became the template for later work.
